# skill: review

テスト・レビュー規約に基づいて実装後のコードをレビューする。

## 使い方

```
/review <対象機能名 or ファイルパス>
例: /review EventService
例: /review app/core/event_generation.py
```

## レビュー観点（汎用）

### 1. API / インターフェース設計
- [ ] 入出力の型・スキーマが定義されているか
- [ ] エラー時に適切なステータス・メッセージを返すか
- [ ] パラメータの型・必須/任意が正しいか

### 2. ドメインロジック
- [ ] 責務が単一になっているか（層をまたぐ依存がないか）
- [ ] 外部依存（HTTP / DB クライアント）がドメイン層に漏れていないか
- [ ] 型ヒント / 型注釈が付いているか

### 3. データ操作
- [ ] トランザクション（コミット・ロールバック）が適切か
- [ ] N+1 クエリなど非効率がないか
- [ ] リソース（接続・ファイル）が確実に解放されるか

### 4. 外部 API 連携
- [ ] タイムアウト設定があるか
- [ ] リトライ処理があるか
- [ ] エラーレスポンスが適切にハンドリングされているか

### 5. テスト
- [ ] 正常系テストがあるか
- [ ] 異常系テストがあるか（不正入力・外部 API エラー）
- [ ] 外部依存がモック化されているか

### 6. harness / eval / registry（本リポジトリ固有）
- [ ] `registry/models.yaml` の revision がピン留めされているか
- [ ] `results/results.csv` への追記が既存行を書き換えていないか（追記のみ）
- [ ] eval設定（promptfoo/DeepEval等）のゴールデンセット版が固定されているか
- [ ] `datasets/golden/` に実データ（samples以外）を含めていないか
- [ ] 重要な決定が ADR (`docs/adr/`) に残っているか

## 出力形式

```
[API] エラー時に詳細メッセージが空 → 修正必要
[Core] ドメイン層が HTTP クライアントを import → 層を分離
[DB] N+1 クエリの可能性 → eager loading を検討
[Test] 異常系テストなし → 追加必要
```

> 言語固有の規約レビューが必要な場合は、別途 `py-review` 等の言語別スキルを併用する。

## 実行後の改善確認（必須）

スキル実行の最後に、次を必ず人間へ確認する。

1. 今回の進め方の感想（良かった点）
2. 使いにくかった点・迷った点（使い勝手）
3. エージェントからの改善提案（手順 / コマンド / 出力）
4. このスキルを今すぐ更新するか（Yes / No）

### 遷移ルール

- Yes: `/update-skill review` を実行し、改善案を提示して承認後に反映する
- No: 更新見送り理由を 1 行で記録し、次回見直しの条件を確認する
