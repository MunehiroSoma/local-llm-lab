## 1. 背景・目的

個人所有の4台のハードウェア群（Mac M4 Pro / RTX 5060 Ti / RTX 5070 / DGX Spark）を使い、
ローカルLLM・マルチモーダル(omni)推論の「フレームワーク選定 → モデル比較 → PoC → 継続評価」を
自動化・再現可能な形で行うラボ環境を構築する。

主要な課題:
- 複数ハード間でフレームワーク・モデルの組み合わせが爆発的に多く、属人的な印象論になりがち
- omni（音声・動画・テキスト入出力）対応モデルの評価基盤が存在しない
- モデル入れ替えのたびに評価設定を書き直す非効率さを排除したい

## 2. スコープ

### 対象（やること）

- フレームワーク選定: vLLM（CUDA）/ MLX-VLM（Apple Silicon）/ Ollama（共通baseline）
- モデル比較: テキスト・コーディング・マルチモーダル・omni の4カテゴリ
- 4層評価: Fit（搭載可否）→ Speed（tok/s）→ Capability（標準ベンチ）→ Task（自前データ）
- 継続評価: `results/results.csv` への追記とレポート生成
- ハーネス実装: OpenAI互換 API 経由で全ランタイムを均一測定

### 対象外（やらないこと）

- クラウドLLM（OpenAI/Claude API等）の比較
- モデルのファインチューニング・学習
- BtoC/BtoB 向けサービスとしての公開
- WebUI の実装（評価結果はCSV/Markdownレポートのみ）

## 4. 機能要件

| ID | 機能 | 概要 | 優先度 | 受け入れ条件 |
|---|---|---|---|---|
| FR-01 | モデルレジストリ | `registry/models.yaml` でモデルID・ランタイム・対応環境を一元管理 | S | yaml load が正常完了 / pre-commit validate-registry 通過 |
| FR-02 | Fit テスト | モデルをロードし VRAM/RAM に収まるか確認し結果を記録 | S | `results.csv` に fit=True/False が追記される |
| FR-03 | Speed ベンチ | 単一ストリームの tok/s・TTFT を計測 | S | `results.csv` に tok_s/ttft_ms が追記される |
| FR-04 | 標準ベンチ連携 | lm-evaluation-harness / llm-jp-eval を呼び出しスコアを記録 | A | `results.csv` に std_bench が追記される |
| FR-05 | Task 評価 | promptfoo/deepeval で自前ゴールデンセットを使い task_score を記録 | A | JSON schema 検証 + LLM-rubric スコアが結果に残る |
| FR-06 | omni 評価 | 音声・動画入力を含む omni 評価ケースを実行 | B | DGX Spark / RTX 5060 Ti 上で omni パイプラインが動作する |
| FR-07 | レポート生成 | `results/reports/` に Markdown/HTMLレポートを自動生成 | B | モデル間比較表・グラフが出力される |
| FR-08 | モデルオンボーディング | `make onboard MODEL=<id>` で 4 層評価を一括実行 | B | 1コマンドで FR-01〜FR-04 が順次実行される |

## 5. 非機能要件

| 区分 | 要件 |
|---|---|
| 再現性 | `registry/models.yaml` の `revision` を固定し、同一コミット・同一ハードで結果が再現できる |
| セキュリティ | `datasets/golden/` の実データはコミット禁止（公開リポ）。guard_golden.sh + CI で二重保護 |
| 可観測性 | 全測定結果は `results/results.csv` に追記（書き換え禁止）。生ログは `results/raw/` |
| 移植性 | OpenAI互換 API を唯一インターフェースとし、ランタイム差異をハーネスから隠蔽（ADR 0002） |
| 保守性 | 新モデル追加 = `models.yaml` に1行 + `make onboard` のみ（ADR 0002 §13.5） |
| テスト | Python コードは pytest -q で実行可能。pre-commit + mypy strict + ruff で品質担保 |

## 6. 制約・前提

- 実行環境: 4台のローカルマシン（`registry/hardware.yaml` 参照）
- 技術スタック: Python 3.11 / uv / ruff / mypy / pytest / Docker CLI（envs/）
- ランタイム: vLLM (CUDA) / MLX-VLM (Apple Silicon) / Ollama (共通)
- AI-DLC: Inception → Construction → Operations の3フェーズ、各遷移に人間承認ゲート（ADR 0005）
- ブランチ戦略: GitHub Flow + squash merge（pre-commit + CI 通過必須）

## 7. 用語定義

| 用語 | 意味 |
|---|---|
| BOLT | 短命ブランチ（数時間〜1日）。Unit of Work の実装単位 |
| 4層評価 | Fit → Speed → Capability → Task の順に評価する本ラボの標準フロー |
| omni | 音声・動画・テキストを入出力できるマルチモーダルモデル |
| tok/s | tokens per second。単一ストリームの生成速度 |
| TTFT | Time To First Token。レイテンシ指標 |
| ゴールデンセット | 手動で正解ラベルを付けた評価データ（実仕様書等）。公開リポにはコミット禁止 |
| ADR | Architecture Decision Record。重要決定を文書化したもの |
