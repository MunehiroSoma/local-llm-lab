# ADR 0006: AIコードレビューの多層観点アーキテクチャ構想（単独開発向け・将来検討）

- ステータス: accepted
- 日付: 2026-07-01
- 実装: Issue #39 にて、導入コストが低く効果が見込める以下の項目のみ `review` スキール /
  `docs/coding-standards/review-checklist.md` へ反映済み（観点カテゴリ分割・並列サブエージェント起動・
  ノイズ抑制ルール・読み取り専用権限・モデル適材適所・出力フォーマット統一・PR作成時限定トリガーの明記）。
  GitHub連携（インラインコメント）・複数モデル一致度・技術領域別振り分け・ship実行時の自動起動フックは
  本ADRの決定事項として記録するが未実装であり、コスト対効果を見て別Issueで検討する。

## 背景 / 課題
`review` スキルは現状、単一エージェントが `docs/coding-standards/review-checklist.md` の
全観点（API/ドメイン/データ/外部API/テスト、+harness/eval/registry）を一度に見る構成。
以下の参考記事はいずれも「1エージェントに全観点を見せると精度が落ちる」という共通の
問題意識から、観点分割・複数エージェント構成に至っている。単独開発ラボでもこの発想の
一部は転用価値があるため、方針を先に記録しておく。

## 決定
記事の3層構成を、本ラボ（単独開発・Claude Code・GitHub Flow）向けに以下へ写像する。
**現時点では概念のみ記録し、実装は行わない**（別途 Issue化してから着手）。

| 記事の層 | 本ラボでの対応 |
|---|---|
| 第1層（親エージェント / オーケストレーション） | `review` スキル本体。変更ファイルを技術領域（Python harness / eval設定 / registry yaml / docs）で振り分け、該当観点のみ起動する |
| 第2層・静的解析エージェント | 既存の pre-commit（ruff/mypy）で代替。AIエージェント化しない（自動化済みのため不要） |
| 第2層・コンテキスト収集 | 既存PRコメント・関連ADR/研究ノートを読み、「指摘済み」「対応不要判定済み」を先に分類してから本レビューへ渡す |
| 第2層・レビュー精査（ゲートキーパー） | 単独開発でのコスト/複雑度に見合わないため**当面は省略**。観点別レビューの結果をそのまま `review` スキルが集約する |
| 第3層（観点特化の実施エージェント） | `docs/coding-standards/review-checklist.md` のカテゴリ（基本品質/エラー処理/セキュリティ/テスト/設計整合性/harness-eval-registry 等）を1エージェント=1観点に分割し、Explore系サブエージェントで並列実行する候補とする |

### 転用するルール（ノイズ抑制、OpenWork記事の「仕組み1」＋SIOS記事の教訓）
観点特化エージェントを実装する際は次を厳守する:
- 良い点の指摘をしない（指摘のみに集中）
- 修正案の提案はしない（修正は実装者/別スキルの責務）
- YAGNI違反（過剰設計）の指摘はしない
- 差分に含まれない既存実装への指摘はしない
- 観点ごとに入力範囲・合格ラインを絞る（SIOS記事: 「1体に全部やらせると失敗」の教訓。例:
  セキュリティ観点は差分のみ、設計整合性観点はADR/研究ノートも読ませる、のように観点で入力を変える）
- レビューエージェントには **Read/Grep/Glob 相当の読み取り専用権限のみ**を与え、書き込みはさせない（Qiita記事）

### モデル適材適所・起動規模の可変（Qiita記事「専門レビュアー18体」）
- 規約・フォーマット系の機械的チェック（命名規則・import整理・禁止パターン検出等）は軽量モデル（haiku相当）
- 設計判断・harnessロジック・アーキテクチャ整合性等は上位モデル（sonnet/opus相当）
- 変更規模に応じて起動する観点数を可変にする（小さいPR=主要観点のみ、model-onboardingや大規模harness変更=全観点）
  → 単独開発でも無駄なトークン消費を避けつつ精度を確保できるため、当面この2点は積極的に取り込む候補とする

### レビュータイミングの多段化（GENDA記事・Qiita記事）
- 本ラボは既に pre-commit（ruff/mypy）で機械的チェックの大半をカバー済みのため、
  「編集直後の自動レビュー」層は新設しない（オーバーヘッドが単独開発に見合わない）
- AIによる観点別レビューは **PR作成時（`ship`実行後、人間承認前）のみ**に限定してトリガーする
  （GENDA記事の「push毎に実行しない」教訓を踏襲。pre-push相当は既存のpre-commitで代替済み）

### 複数モデル一致度（記事の「仕組み2」）について
記事はClaude/GPTを2体ずつ計4体で多数決を取るが、単独開発ラボでは通常運用コストに見合わない。
- デフォルト: 観点あたり1パスのみ（現行と同等のコスト感）
- 例外: model-onboarding のような重要判断（採用/不採用）やharness中核ロジックの変更時のみ、
  同一観点を条件・温度を変えて2パス実行し、両方が指摘した内容だけ採用する運用を**任意で**許可する

### GitHub連携（OpenWork記事「仕組み3」＋GENDA記事）
- 指摘はPRのインラインコメントとして投稿し、使用したプロンプト/観点も本文に残す（透明性）
- 再レビュー時は前回レビュー以降のコミット差分のみを対象にし、既存コメントを
  「指摘済み」「対応不要」で分類してから重複指摘を避ける
- 1つの長文コメントに全指摘をまとめない（GENDA記事の反省点）。指摘箇所ごとにインラインで分割する

### 出力フォーマットの統一（Qiita記事）
観点特化エージェントの出力は `review` スキル既存の `[カテゴリ] 指摘内容 → 対応` に加え、
`SEVERITY(Must/Should/Nits) / FILE / LINE / ISSUE / 該当観点` を機械可読な形で揃える。
既存の `review-checklist.md` の重要度区分（Must/Should/Nits）とそのまま整合するため追加コストは小さい。

### テスト観点への転用（nexta_記事、`test-check`スキル向け）
「7人の意地悪なQA」の発想は `test-check` の異常系確認に転用できる余地がある。
harness/eval文脈での例: 不正な`registry/models.yaml`設定を渡すユーザー、途中で推論サーバが
落ちる環境、golden実データを誤ってコミットしようとする操作者、等の批判的ペルソナを
異常系チェック項目の発想源として使う（**今回はADR記録のみ、`test-check`への反映は別Issueで検討**）。

## 理由 / 代替案
- 代替案A: いずれかの記事の構成（4層/5専門/18体/3体等）をそのまま実装
  → 却下。単独開発では専任静的解析エージェントの並列化やモデル多重化・大量エージェント常設の
    コスト対効果が低い。ただし「モデル適材適所」「起動規模の可変」「読み取り専用権限」
    「観点ごとの入力範囲を絞る」は単独開発でも導入コストが低く効果が見込めるため個別に採用候補とする。
- 代替案B: 現状の単一エージェントレビューを維持
  → 一旦は許容するが、観点を絞らない一括レビューは各記事が共通して指摘する
    「タスクが大きいと精度低下」の課題をそのまま抱えるため、観点分割だけは早期に取り込みたい。

## 影響
- `docs/coding-standards/review-checklist.md`（観点カテゴリの単位で分割される前提、重要度区分はそのまま流用）
- `.claude/skills/review/SKILL.md`（将来、観点別サブエージェント起動・出力フォーマット統一へ拡張する余地を残す）
- `.claude/skills/test-check/SKILL.md`（批判的ペルソナ発想を異常系チェックに転用する余地、別Issue検討）
- 実装に着手する場合は `create-issue` で計画Issue化し、Inceptionフェーズの承認を経てから行う
